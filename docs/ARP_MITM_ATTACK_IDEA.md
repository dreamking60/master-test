## ARP 欺骗型 MITM：具体思路（原理 + 实验框架）

> 说明：本文用于毕业设计/科研写作与隔离环境下的安全实验设计，聚焦**原理、威胁模型、模块框架、观测指标、结果记录方式**。  

---

## 1. 背景：为什么 ARP 会成为 MITM 的入口

- **ARP 的作用**：在同一二层广播域（同网段/同 VLAN）里，主机需要把“目标 IP”解析成“目标 MAC”，才能把以太网帧发出去。
- **缓存机制**：解析结果会进入 ARP/邻居表（neighbor table），并在一段时间内复用。
- **安全弱点（核心点）**：在很多系统/配置中，主机对 ARP 信息的信任边界较弱，IP→MAC 绑定关系可能在缺乏强认证的情况下被更新或污染。

因此，一旦攻击者能让受害机把“关键 IP（通常是默认网关）”绑定到“攻击者 MAC”，就可能把数据链路层的下一跳“改道”到攻击者，从而具备 MITM 的先决条件。

---

## 2. 威胁模型与前提假设（写进论文很关键）

### 2.1 参与方
- **Victim（受害机）**：机器人 VM 或控制端 VM（取决于你要观察哪边被劫持）
- **Gateway（网关）**：同网段的默认网关/路由器（可能是你实验中的某台 VM）
- **Attacker（攻击机）**：希望进入通信路径的中间节点

### 2.2 前提条件（决定实验是否“可行”）
- **L2 可达**：Victim、Gateway、Attacker 处于同一二层广播域（这是 ARP 相关实验的硬前提）。
- **流量可转发**：若攻击机把自己插入路径但**不具备转发能力**，现象会更像中断/拒绝服务，而非稳定 MITM。
- **目标流量类型**：ROS2/DDS 大量使用 UDP（发现与通信），MITM 的“可观测性”很强；但“可读/可改”取决于是否启用 SROS2（加密/认证）。

### 2.3 你论文里可以明确的“攻击目标”
ARP 欺骗型 MITM 的本质目标并不是“发包攻击 ROS2”，而是：
- **改变路径**：让 Victim 认为“关键 IP 的 MAC = Attacker MAC”
- **维持路径**：抵抗缓存更新/超时带来的回滚
- **保持连通**：通过转发让业务不中断，以证明“中间人”而非“断网”

## 3. 两套实验场景的具体思路（与你的需求对应）

你提到要实现两套：  
1) **在已配置路由转发基础上**进行  
2) **通过欺骗让攻击机成为中间人**

下面按“论文/实验设计”的方式把两套的差别写清楚。

### 场景 A：已有路由转发基础（“转发能力先满足”）

**定位**：你先把 Attacker 变成一个具备转发能力的节点（这点你仓库的 `attack_experiments/mitm_attack` 已经做了“路由型 MITM”框架）。  
ARP 在这里的作用是：**只负责把二层下一跳引到 Attacker**，从而无需在 Victim/Controller 上手动改网关也能“走你这里”。

**你可以在论文里强调的优势**：
- 关键难点（稳定转发/不中断）先解决，实验更稳定，便于复现实验数据。
- 可以做对照：  
  - 改网关（显式重定向）  
  - 不改网关，仅通过 L2 映射变化引入中间人（隐蔽性更强）

**你需要验证的核心现象**（不需要写攻击细节也能成立）：
- Victim 的 neighbor table 中，**网关 IP 的 MAC 发生变化**
- ROS2 侧通信仍保持（节点/话题仍可见），说明转发链路成立

### 场景 B：仅通过欺骗把 Attacker 变成 MITM（“路径先改变，再谈转发”）

**定位**：这更像“纯网络层路径劫持”的实验叙述——Attacker 不靠修改 Victim 的静态网络配置，而是通过改变二层绑定关系进入路径。

**你论文里可以强调的难点/风险**：
- 实验对网络拓扑更敏感（必须同一二层域；VM 网络模式会显著影响结果）。
- 若只影响单向绑定（只骗 Victim 或只骗 Gateway），容易出现“半通/断流”，不容易获得稳定 MITM。

**你需要验证的核心现象**：
- 双向路径改变的证据（至少能在 Victim 或 Gateway 侧观测到关键 IP→MAC 的异常变化）
- 业务可用性（不中断）与异常告警（出现变更事件）的时间线对齐

## 4. 攻击链条抽象成“模块框架”（便于你自己实现 & 写报告）

你可以把 ARP 欺骗型 MITM 抽象为 4 个模块（章节结构清晰，也方便后续做防御）：

### 4.1 环境识别与基线模块（Baseline）
- 输入：接口、受害机/网关 IP、实验角色
- 输出：初始 IP→MAC 基线（可写入 JSON，便于复现实验）
- 记录项：neighbor table 快照、默认网关信息、时间戳

### 4.2 路径影响模块（Path Manipulation）
- 目标：让关键 IP 的 L2 绑定发生偏移（从“真实 MAC”变为“攻击者 MAC”）
- 论文表述建议：强调“对缓存机制的影响”和“持续维持”的必要性，而不是具体实现细节

### 4.3 转发/旁路模块（Forwarding / Relay）
- 目标：保证被引流的流量可继续到达真实目的地（从而形成 MITM 而非中断）
- 可对照实验：  
  - 无转发：连接中断/话题消失  
  - 有转发：连接保持但路径证据出现

### 4.4 恢复与清理模块（Cleanup）
- 目标：实验结束恢复正常绑定，避免污染影响下一轮实验数据
- 验证：neighbor table 回到 baseline 或稳定状态

## 5. 你该观测什么（用数据支撑“我真的做成了 MITM”）

建议固定一组指标，做“场景 A vs 场景 B”的对照表：

### 5.1 二层证据（必须）
- **IP→MAC 变更时间线**：尤其是默认网关 IP 对应的 MAC
- **变更频率**：是否抖动/反复切换（体现实验稳定性）

> 本仓库已提供防御侧取证工具：`attack_experiments/defense_test/scripts/monitor_arp.py`  
> 它能把 baseline、变更告警、日志落到 `defense_test/results/`，适合直接当实验附件。

### 5.2 连通性与业务侧证据（建议）
- ROS2 节点发现是否持续
- 关键话题（如 `/cmd_vel`）是否持续可见
- 控制时延/抖动的主观或客观记录（可做定性 + 定量）

### 5.3 安全机制对比（论文加分点）
- **不开 SROS2**：中间人可能“可读/可改”
- **开 SROS2**：内容层面不可读不可改，但仍可能“可阻断/可降级”

## 6. 防御与检测（可写成你项目的贡献点）

### 6.1 检测思路（你可以写“我做了监控并验证”）
- neighbor table 监控：网关 IP→MAC 变更告警（高信号）
- MAC collision 观察：多个 IP 映射到同一 MAC（弱信号，作为上下文）
- ROS2 侧监控：异常发布者/异常频率（你仓库已有 node 监控与频率监控）

### 6.2 防御思路（写“工程建议”很适合毕业设计）
- 二层隔离：把机器人网络与非受信网络隔离（VLAN/物理隔离）
- 端口与访问控制：DDS 端口防火墙、VPN 访问替代裸露局域网访问
- 端到端安全：SROS2（加密 + 认证）防止内容层面被篡改
- 运行时监测：把 ARP 监控与 ROS2 监控结合成自动告警

---

## 7. 写作建议：把“两个场景”写成对照实验

建议你的实验结果章节按下面结构写：
- **实验设置**：拓扑图、VM 网络模式、角色 IP、是否启用 SROS2
- **场景 A**：路径证据（ARP 变更）+ 通信保持（ROS2 可用）+ 监测日志
- **场景 B**：同样的证据链 + 稳定性对比（是否更易抖动/失败）
- **结论**：  
  - 哪种场景更稳定/更隐蔽/更易检测  
  - 防火墙/隔离/SROS2 对两类场景的影响差异
